# Backend API Integration Blueprint & Hand-off Guide

This guide is designed for the Backend Developer to connect the frontend UI with dynamic backend REST APIs. 

Currently, the frontend uses a **High-Fidelity Mock Service Layer (`localStorage`)** to simulate form submission, admin approval control, search filtering, and individual partner detail rendering. Swapping this simulated layer with your live REST APIs will require implementing the endpoints and schemas detailed below.

---

## 1. Core Database Schemas & Types

The frontend is built using TypeScript with strict interfaces. Your database records should map directly to the following types:

### A. Partner Entity Schema
This is the primary companion profile record.

| Field Name | Type | Description |
| :--- | :--- | :--- |
| `id` | `String` (Primary Key) | **Unique SEO Slug** (e.g. `divya-sharma-123456`). Generated by combining the name-slug with a unique ID or database sequence. |
| `name` | `String` | Full name of the partner. |
| `age` | `Number` | Age of the partner (minimum 18). |
| `gender` | `String` | e.g. `"Female"`, `"Male"`, `"Non-Binary"`. |
| `bio` | `String` | Personal description or introduction. |
| `location` | `String` | City/State representation (e.g., `"Mumbai, India"`). |
| `rating` | `String` (or `Decimal`) | Rating from 1.0 to 5.0 (default `"4.9"` for new partners). |
| `verified` | `Boolean` | Flag indicating if the admin has reviewed and approved the profile. |
| `distance` | `String` | Mock distance calculation from the searcher (e.g., `"0.8 km"`). |
| `image` | `String` (URL) | Public URL of the profile avatar image. |
| `reviews` | `Array<Review>` | Sub-entity array of reviews left by users (see below). |
| `gallery` | `Array<Gallery>` | Sub-entity array of gallery images uploaded by the companion (see below). |
| `pricing` | `Pricing` (Nested Object) | Hourly rate packages (see below). |

### B. Sub-Entity Schemas

#### `Pricing` (Hourly Rates)
```typescript
interface Pricing {
  oneHour: number;     // e.g. 500
  twoHours: number;    // e.g. 900 (calculated/discounted)
  threeHours: number;  // e.g. 1250
  fourHours: number;
  fiveHours: number;
  eightHours: number;
}
```

#### `Gallery` (Portfolio Photos)
```typescript
interface Gallery {
  id: string;          // Image primary key
  image: string;       // Public URL of the gallery photo
}
```

#### `Review` (Customer Feedback)
```typescript
interface Review {
  id: string;
  name: string;        // Reviewer's name
  role: string;        // e.g., "Verified Client"
  text: string;        // Feedback content
  image: string;       // Reviewer's avatar URL
  rating: number;      // 1 to 5
}
```

## 2. API Endpoints Required

### 🟢 1. Register Companion Profile
*   **Endpoint:** `POST /api/partners/apply`
*   **Content-Type:** `multipart/form-data`
*   **Description:** Triggered when a companion submits the "Become a Partner" form.
*   **Payload Requirements:**
    *   `fullName`: String
    *   `age`: Integer
    *   `gender`: String
    *   `city`: String
    *   `mobile`: String
    *   `email`: String
    *   `bio`: String
    *   `hourlyRate`: Decimal
    *   `photo`: File upload (Profile avatar)
    *   `gallery`: Array of Files (Up to 9 companion gallery photos)
    *   `idProofs`: Array of Files (ID Card Front, ID Card Back, and Selfie with ID)
    *   `languages`: Array of Strings
    *   `availability`: Array of Strings (e.g., `["Monday", "Tuesday"]`)
*   **Server-Side File Upload Recommendation:**
    *   We recommend processing these files using a cloud storage bucket (like **AWS S3** or **Cloudinary**).
    *   Store the public read-only CDN URLs (e.g. `https://s3.amazonaws.com/...`) under the database columns for `image`, `gallery[].image`, and `idProofs`.

---

### 🟢 2. Fetch Companion Grid (Browse & Home Page)
*   **Endpoint:** `GET /api/partners`
*   **Query Filters:**
    *   `age`: e.g. `20-30` or `30+`
    *   `location`: e.g. `Mumbai` (Partial text match)
    *   `rating`: e.g. `4.5` (Minimum rating threshold)
    *   `gender`: e.g. `Female`
    *   `priceMin` / `priceMax`: e.g. `300` / `800` (Filters pricing range mapping to `pricing.oneHour`)
    *   `verified`: e.g. `true` (Browse grid should default to loading only verified partners)
*   **Expected Response:** JSON array of `Partner` objects conforming to the schema in Section 1.

---

### 🟢 3. Fetch Single Companion Profile (Dynamic Details Page)
*   **Endpoint:** `GET /api/partners/:id` (where `:id` can be either the primary key or unique name-slug)
*   **Description:** Loads companion details on `/partners/[id]`.
*   **Expected Response:** A single `Partner` entity object. Returns `404` if not found.

---

### 🟢 4. Admin Verification Control Center (Application Pipeline)
*   **Endpoint:** `PATCH /api/partners/applications/:applicationId/status`
*   **Description:** Handles verification status transitions in the Admin Panel.
*   **States Lifecycle:**
    *   `DRAFT` (Initial application state)
    *   `PENDING` (Submitted, waiting for admin review)
    *   `VERIFIED` (Approved, profile injected live into `/browse-partners` grid)
    *   `REJECTED` (Declined profile application)
    *   `NEEDS_REVISION` (Sent back to applicant for revision, with comments)
*   **Payload Requirements:**
    *   `status`: `"VERIFIED"` | `"REJECTED"` | `"NEEDS_REVISION"`
    *   `notes`: String (Revision comments if status is `"NEEDS_REVISION"`)

## 3. Frontend-to-Backend Connection Steps

Once your REST APIs are ready, the frontend developer only needs to update **one service file** to transition:

📁 **Target File:** `src/modules/partner/services/partner.service.ts`

### Current Simulation Code:
```typescript
export const PartnerService = {
  async getPartners(): Promise<Partner[]> {
    // 1. Fetch backend mock list
    let remotePartners = await fetchPartnersListMock();
    
    // 2. Reconcile with local approved storage
    let localApproved: Partner[] = [];
    if (typeof window !== "undefined") {
      const saved = localStorage.getItem("approved_partners");
      if (saved) { ... }
    }
    return [...localApproved, ...remotePartners];
  }
}
```

### Future Production API Integration:
Simply replace the internal method queries with database fetches:
```typescript
export const PartnerService = {
  async getPartners(filters?: any): Promise<Partner[]> {
    const queryParams = new URLSearchParams(filters).toString();
    const response = await fetch(`https://api.yourdomain.com/partners?${queryParams}`);
    if (!response.ok) throw new Error("Failed to load partners");
    return await response.json();
  },

  async getPartnerById(id: string): Promise<Partner> {
    const response = await fetch(`https://api.yourdomain.com/partners/${id}`);
    if (!response.ok) throw new Error("Partner not found");
    return await response.json();
  }
}
```
